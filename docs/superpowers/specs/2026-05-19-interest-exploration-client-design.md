# 兴趣探索订阅端 v1 设计文档

## 1. 背景

基于 `PRD-兴趣探索订阅端v1.0.md` 与 `原型.html`，首期开发目标是从空仓快速搭建一个可运行、可继续扩展的 MVP 工程。产品面向移动端，客户端采用 React Native + Expo，服务端采用 FastAPI，支持“领域选择、今日简报、陌生抽卡、收藏回看、站内阅读”的完整最小闭环。

本设计文档聚焦首期工程落地方式，不覆盖真实抓取、聚合、去重算法的完整实现细节，而优先定义清晰的工程边界、数据契约和页面流转，确保前后端可以并行开发。

## 2. 目标与范围

### 2.1 本期目标

- 建立单仓工程结构，支持前后端并行开发
- 用 Expo 实现与原型一致的核心移动端体验
- 用 FastAPI 提供可联调的接口骨架与 mock 数据
- 用共享数据模型统一领域、条目、抽卡等结构
- 跑通从选择领域到阅读完成的用户主链路

### 2.2 本期范围

- Onboarding 领域选择
- 今日简报矩阵浏览
- 阅读页 Modal
- 抽卡页
- 收藏页
- 设置页基础入口
- 本地存储
- FastAPI mock 接口

### 2.3 明确不在本期实现

- 真正的登录与跨设备同步
- 真实抓取、抽取式结论卡、复杂去重折叠算法
- 推送与通知
- Web 端

## 3. 方案结论

采用单仓结构：

- `apps/mobile`：Expo 移动端应用
- `apps/api`：FastAPI 服务
- `packages/shared`：共享类型、常量、mock 数据、接口 schema
- `docs`：设计与规划文档

采用该结构的原因：

- 与 PRD 已确认技术选型一致
- 适合空仓从 0 初始化
- 方便使用 subagent 并行推进前端、后端、共享模型
- 降低前后端对字段理解不一致的风险

目标工作分支命名为 `dn_0.1_20260519`。

该包的目标是让移动端与服务端围绕同一份领域和数据约束开发。

## 5. 前端信息架构与页面设计

### 5.1 导航结构

首期按 PRD 采用 3 个底部 Tab：

- 今日
- 抽卡
- 收藏

全局增加：

- 首次进入的 Onboarding 流程
- 阅读 Modal
- 设置入口

### 5.2 Onboarding

目标：

- 用户首次进入后快速选择 1 至 5 个领域
- 默认建议 3 个领域
- 完成后进入今日简报页

交互规则：

- 最少选择 1 个领域才允许继续
- 超过 5 个领域时阻止选择并提示原因
- 首期只做本地存储，不做账号绑定

### 5.3 今日简报页

目标：

- 保留原型中的“纵向领域 + 横向条目”的矩阵式浏览
- 提供清晰的进度反馈和结束感

页面组成：

- 顶部标题与总进度
- 纵向领域容器
- 每个领域下横向 10 条 slide
- 每条 slide 包含：领域标签、条目序号、标题、来源/时间、结论卡、阅读全文按钮、标记已读按钮

首期实现要求：

- 优先保证交互稳定而非完全还原原型动画
- 支持进入阅读页时自动记为已读
- 全部处理完成后展示“今日已完成”状态卡

### 5.4 阅读页

形态：

- 使用 Modal 或全屏覆盖层

展示内容：

- 标题
- 来源与时间
- 结论卡
- 正文内容或抓取失败提示
- 打开原文按钮

错误降级：

- 若全文获取失败，页面不白屏
- 提示“站内阅读暂时不可用”并允许跳转原文

### 5.5 抽卡页

目标：

- 展示每日 3 张陌生领域卡
- 每张卡必须能解释为什么出现

卡片包含：

- 陌生领域名称
- 关联解释文案
- 代表性标题预览
- 加入关注按钮
- 不感兴趣按钮

交互规则：

- 加入关注后，如果未满 5 个领域，立即更新今日页数据
- 如果已满 5 个领域，则给出友好限制提示

### 5.6 收藏页

首期能力：

- 本地收藏列表
- 按标题/来源搜索
- 按领域过滤的结构预留

### 5.7 设置页

首期先做轻量内容：

- 领域管理入口
- 排序偏好占位
- 清缓存
- 隐私与数据说明占位

## 6. 前端状态与本地存储设计

本地存储优先使用 `AsyncStorage`。

建议存储键：

- `selected_domains`
- `read_item_ids_{date}`
- `favorite_item_ids`
- `cached_digest_{date}`
- `cached_article_{item_id}`
- `explore_seen_{date}`
- `theme_mode`

前端状态分层：

- `UI 状态`：当前 tab、Modal 开关、手势位置
- `业务状态`：领域、今日简报、抽卡、收藏、已读
- `缓存状态`：接口返回结果、本地恢复内容

建议使用轻量全局状态管理配合请求缓存，避免页面各自维护重复数据。

## 7. 后端接口设计

首期 FastAPI 先提供 4 组接口。

### 7.1 领域列表

- `GET /domains`
- 返回首发领域列表与相邻关系基础信息

### 7.2 今日简报

- `POST /digest/today`
- 入参：已选领域、排序偏好、日期
- 返回：每个领域最多 10 条条目

### 7.3 抽卡

- `POST /explore/draw`
- 入参：已选领域、最近已抽卡记录、日期
- 返回：3 张抽卡结果与推荐原因

### 7.4 文章详情

- `GET /items/{item_id}`
- 返回：标题、来源、时间、结论卡、正文、原文链接、抓取状态

### 7.5 首期实现策略

- 接口先返回 mock 数据
- 字段结构提前按真实场景设计
- 服务层预留真实聚合、抓取、去重接入位

## 8. 共享数据模型

共享层至少定义以下实体：

- `Domain`
- `DigestItem`
- `DigestGroup`
- `ExploreCard`
- `ArticleDetail`
- `SortPreference`

基本原则：

- 所有字段优先满足移动端展示需要
- 服务端 schema 与前端类型尽量同名同义
- mock 数据直接复用共享结构生成

## 9. 核心数据流

### 9.1 首次使用

1. 客户端读取本地已选领域
2. 若为空则进入 Onboarding
3. 用户完成选择后写入本地
4. 客户端请求今日简报接口
5. 返回结果写入本地缓存并渲染

### 9.2 日常打开

1. 读取当日缓存
2. 若缓存存在则先展示
3. 后台刷新今日简报
4. 同步更新已读进度和界面

### 9.3 阅读动作

1. 用户点开条目
2. 客户端请求文章详情
3. 成功则展示正文
4. 失败则展示降级提示和原文入口
5. 打开阅读页时将条目标记为已读

### 9.4 抽卡关注

1. 用户在抽卡页点击加入关注
2. 客户端先校验是否已满 5 个领域
3. 成功后更新本地领域集合
4. 重新请求今日简报
5. 今日页刷新并展示新增领域

## 10. 错误处理与降级策略

### 10.1 客户端

- 接口失败时保留旧缓存，避免空白页
- 当日简报不足 10 条时展示明确提示，不凑数
- 抽卡不足 3 张时展示原因并用安全领域补位

### 10.2 服务端

- mock 阶段所有接口必须保证可返回结构完整的数据
- 真实抓取接入后，抓取失败不能影响条目基础展示
- 正文不可用时统一回退原文链接

## 11. 测试策略

首期测试以高价值检查为主，不追求大而全。

移动端优先验证：

- 领域选择上下限
- 今日进度与已读更新
- 阅读页打开与降级
- 抽卡加入关注后的刷新
- 收藏与搜索基础流程

后端优先验证：

- 接口响应结构稳定
- 每个接口返回字段完整
- 错误状态下仍符合约定响应

## 12. 并行开发拆分

建议按以下方向并行：

### 12.1 子任务 A：移动端基础骨架

- 初始化 Expo 工程
- 建立主题、导航、页面目录
- 搭建底部 Tab 与阅读 Modal 基础结构

### 12.2 子任务 B：后端接口骨架

- 初始化 FastAPI 工程
- 建立路由、schema、服务层目录
- 输出可联调 mock 接口

### 12.3 子任务 C：共享模型

- 定义领域枚举
- 定义 DTO
- 生成 mock 数据
- 提供前后端共用常量

### 12.4 子任务 D：核心链路联调

- 打通 Onboarding -> 今日 -> 阅读
- 打通 抽卡 -> 加入关注 -> 今日刷新
- 打通 收藏写入与展示

## 13. 风险与决策

### 13.1 矩阵滑动实现复杂

决策：

- 首期以“可用、流畅、清晰”优先
- 若双向手势实现复杂度过高，可先采用分页容器组合实现稳定版本

### 13.2 后端真实能力未完成

决策：

- 首期所有真实抓取、聚合、去重能力用服务骨架与 mock 数据替代
- 不阻塞前端验证体验

### 13.3 游客模式数据易丢失

决策：

- 维持 PRD 约定
- 在设置或文案中明确说明本地存储限制

## 14. 实施前置结论

在进入代码实现前，按以下顺序推进：

1. 基于本设计写实施计划
2. 明确子任务拆分与执行顺序
3. 再开始脚手架初始化与并行开发

当前设计已确认以下关键决策：

- 技术栈采用 React Native + Expo 与 FastAPI
- 仓库结构采用单仓多应用
- 首期前后端一起起步
- 分支命名采用 `dn_0.1_20260519`
